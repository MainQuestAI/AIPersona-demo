# AIpersona Demo 开发简报

日期：2026-04-03  
口径：面向比稿阶段截止目标  
结论状态：`主舞台已成立，但最后一公里仍未完全收口`

## 1. 一句话结论

如果按“能否支撑老板对外比稿演示”来判断，当前完成度大约在 `80%~85%`。  

如果按“Runtime-first 产品 MVP 是否已经成立”来判断，当前完成度大约在 `35%~40%`。

这两个数字不冲突：

- 前者看的是 `前台说服力`
- 后者看的是 `完整产品能力`

当前项目已经明显跨过“PPT Demo / 静态壳”的阶段，进入了“有真实运行时主链、可以演示真实状态流转”的阶段。

---

## 2. 截止目标完成情况

### 2.1 已经完成的高价值部分

#### A. 前台主舞台已经成立

当前已经具备：

- `Workbench / Compare / Twins` 三个一级页面
- 统一视觉底座、导航、状态上下文
- 可直接展示研究主线、概念对比、Twin 资产来源
- 结果区已经具备 `Pitch Script / Recommendation / Evidence / Replay` 入口

业务意义：

- 老板已经不是在展示一个聊天工具
- 已经可以展示“研究如何启动、如何审批、如何执行、为什么得出当前推荐”

#### B. Runtime 最小闭环已经成立

当前已经真实打通：

- 创建 `study`
- 生成 `study_plan_version`
- 提交审批
- 批准 plan
- 启动 `study_run`
- 进入 `mid-run approval`
- `resume` 后完成到 `succeeded`

业务意义：

- 这不是前端假状态
- 这是一条真实写入 PostgreSQL、真实经由 Temporal 的运行时主链

#### C. 前后端已经开始同一条故事线

当前不是“前端讲一套、后端跑一套”，而是：

- Workbench 读取真实 Runtime 投影
- Compare 基于真实 Plan / Run 做对比
- Twins 基于真实 twin payload 和 stimulus payload 做 registry 展示

业务意义：

- 比稿时“可信度”明显提升
- 后续再扩展不会从纯 mock 壳反向重构

## 4. 当前最强的业务价值

截至现在，这个项目最强的地方已经不是“页面好看”，而是以下三点：

### 4.1 可以讲“真实研究正在发生”

现在可以演示：

- 计划如何被生成
- 为什么需要审批
- run 如何启动
- 为什么会中途进入 review
- 为什么 resume 后能继续完成

这会显著提升客户对系统严肃性的判断。

### 4.2 已经具备“为什么值得相信”的雏形

现在不只给出推荐，还能展示：

- winner
- confidence
- compare path
- twins provenance
- trust / replay 入口

这对比稿很关键，因为客户通常不只看答案，更看答案是怎么来的。

### 4.3 已经保住 Runtime-first 路线

当前开发没有退化成“Prompt-first 单轮助手”或“聊天壳 + 函数调用拼装器”。

这意味着：

- 后续能接审批
- 能接恢复
- 能接回放
- 能接评测
- 能接治理

这才是产品竞争力真正能累积的前提。

---

## 5. 当前未收口的问题

### 5.1 比稿最后一公里还没有完全封口

仍然缺：

- 更稳定的浏览器级全链路 smoke
- 老板进入最新 Study 的零摩擦入口完全验证
- 演示路径的最终顺滑化收口

影响：

- 不影响继续开发
- 但会影响“现场演示是否足够稳”

### 5.2 前台仍有少量非业务噪音

当前仍存在：

- React Router future warning
- SSR 场景下的 `useLayoutEffect` warning

影响：

- 不影响功能成立
- 但会影响演示洁净度和工程完成感

### 5.3 产品级能力还远没完成

目前还没有真正进入：

- Asset ingestion
- Twin center 管理能力
- Qual engine
- Quant engine
- Replay/report 正式产物
- Calibration lite
- Security / audit / release hardening

影响：

- 当前适合比稿推进
- 但还不适合把它当作完整 MVP 对外承诺
